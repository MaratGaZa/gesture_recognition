<!--# Руководство по выполнению задач из plan.md-->

## 1. Подготовка
1. Откройте каталог **/docs** и внимательно изучите все файлы:
   - `architecture.md`
   - `MVP_Functions.md`
   - `Code_Quality.md`
   - `stack.md`
2. Сформируйте в голове полное представление о потоке данных, требуемых модулях и принципах качества кода.

## 2. Выбор задачи
1. Откройте `plan.md`.
2. Найдите первую задачу со статусом **TODO** (строка `**Статус**: TODO`).
3. Скопируйте номер задачи (например, `### 3. Реализация модуля Loop`).

## 3. Перевод статуса в *In Progress*
1. В том же файле `plan.md` замените строку  
   `**Статус**: TODO` → `**Статус**: In Progress`.
2. Сохраните изменения и сделайте коммит, например:  
   `git add plan.md && git commit -m "Start task 3 – Loop module"`.

## 4. Выполнение задачи
Для каждой задачи выполняйте следующие подпункты:

| Шаг | Описание |
|-----|----------|
| **4.1** | Прочитайте описание задачи и **Definition of Done**. Убедитесь, что понимаете, какие функции/классы нужно создать и какие тесты написать. |
| **4.2** | Откройте соответствующий файл‑шаблон в `src/…` (если он ещё не существует – создайте его). |
| **4.3** | Реализуйте код, соблюдая принципы из `Code_Quality.md` (единственная ответственность, отсутствие логики в `loop`, работа с DOM только в `render/` и т.д.). |
| **4.4** | Добавьте константы в отдельный файл (`src/constants.js`) при необходимости, чтобы избежать «магических» чисел. |
| **4.5** | Напишите unit‑тесты (Jest/Playwright) для всех публичных функций/классов задачи. |
| **4.6** | Запустите линтер (`npm run lint`) и тесты (`npm test`). Исправьте возникающие ошибки. |
| **4.7** | Выполните ручную проверку в браузере: откройте `index.html`, проверьте, что новая функциональность работает без консольных ошибок. |
| **4.8** | При необходимости профилируйте производительность (FPS, задержка) и доведите её до заявленных значений. |

## 5. Завершение задачи
1. Когда **Все пункты из *Definition of Done* выполнены**, измените статус в `plan.md` на **Done**:  
   `**Статус**: Done`.
2. Сохраните файл и закоммитьте изменения, например:  
   `git add src/ … plan.md && git commit -m "Finish task 3 – Loop module"`.
3. При работе в ветке создайте Pull Request, назначьте ревьюеров и пройдите проверку качества (ESLint, тесты, код‑ревью).

## 6. Переход к следующей задаче
1. Вернитесь к пункту 2 и повторите процесс для очередной задачи со статусом **TODO**.
2. При необходимости, создавайте отдельные ветки для каждой задачи, чтобы облегчить ревью и откат изменений.

## 7. Общие рекомендации
- **Коммиты мелкими**: каждый логический шаг (например, «add debounce util», «write tests for debounce») – отдельный коммит.
- **Документируйте** публичные API с помощью JSDoc или аналогов.
- **Обновляйте README** при добавлении новых модулей или зависимостей.
- **Не забывайте** о глобальном обработчике ошибок (см. задача 11) – приложение не должно падать при отказе камеры или модели.
- **Следите** за метриками производительности: `npm run perf` (скрипт, который выводит FPS и латентность) должен показывать ≥ 20 FPS и < 100 мс задержки.
